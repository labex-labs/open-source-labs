# To panic or Not to panic

Alors, comment décider quand il faut appeler `panic!` et quand il faut retourner `Result`? Lorsque le code panique, il n'y a pas de moyen de récupérer. Vous pourriez appeler `panic!` pour n'importe quelle situation d'erreur, que ce soit une situation où il est possible de récupérer ou non, mais alors vous prenez la décision qu'une situation est irréparable au nom du code appelant. Lorsque vous choisissez de retourner une valeur `Result`, vous donnez des options au code appelant. Le code appelant pourrait choisir d'essayer de récupérer d'une manière appropriée à sa situation, ou il pourrait décider qu'une valeur `Err` dans ce cas est irréparable, donc il peut appeler `panic!` et transformer votre erreur récupérable en une erreur irréparable. Par conséquent, retourner `Result` est un bon choix par défaut lorsque vous définissez une fonction qui peut échouer.

Dans des situations telles que des exemples, du code de prototype et des tests, il est plus approprié d'écrire du code qui panique plutôt que de retourner un `Result`. Explorons pourquoi, puis discutons de situations dans lesquelles le compilateur ne peut pas dire que l'échec est impossible, mais que vous, en tant qu'être humain, le pouvez. Le chapitre se conclura avec quelques directives générales sur la manière de décider si on doit paniquer dans le code de bibliothèque.
